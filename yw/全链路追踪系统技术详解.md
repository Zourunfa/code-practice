# 全链路追踪系统技术详解

## 一、系统概述

### 1.1 背景与目标

在互联网医院移动开方系统中，一个完整的业务流程涉及多个环节：从用户进入、填写问诊资料、医生匹配、问诊开方到最终处方生成。为了实现对整个业务流程的监控、分析和优化，我们构建了一套完整的全链路追踪系统。

**核心目标：**
- 实现业务流程的全链路可追溯
- 提供精准的性能分析和瓶颈定位
- 支持用户行为分析和业务优化
- 保障系统稳定性和问题快速定位

### 1.2 技术架构

全链路追踪系统采用三层架构设计：

```
┌─────────────────────────────────────────────────────────┐
│                     业务层                                │
│  (开方页面、问诊页面、IM聊天等)                            │
└─────────────────────────────────────────────────────────┘
                          ↓
┌─────────────────────────────────────────────────────────┐
│                   追踪管理层                              │
│  ┌──────────────┐  ┌──────────────┐  ┌──────────────┐  │
│  │  TrackStore  │  │ Event Track  │  │  SLS Logger  │  │
│  │  (流程ID)    │  │  (事件埋点)  │  │  (日志上报)  │  │
│  └──────────────┘  └──────────────┘  └──────────────┘  │
└─────────────────────────────────────────────────────────┘
                          ↓
┌─────────────────────────────────────────────────────────┐
│                   数据采集层                              │
│  ┌──────────────┐  ┌──────────────┐                     │
│  │ Sensors SDK  │  │ Aliyun SLS   │                     │
│  │  (神策分析)  │  │  (日志服务)  │                     │
│  └──────────────┘  └──────────────┘                     │
└─────────────────────────────────────────────────────────┘
```



## 二、核心组件详解

### 2.1 TrackStore - 流程ID管理

TrackStore 是全链路追踪的核心，负责生成和管理唯一的流程ID（processId），确保整个业务流程可追溯。

#### 2.1.1 ProcessId 设计

**格式定义：**
```
processId = 门店编码 + YYYYMMDDHHmmss
```

**示例：**
```
F001202602091430
├─ F001: 门店编码
└─ 20260209143045: 2026年2月9日 14:30:45
```

**设计优势：**
- **唯一性保障**：门店编码 + 精确到秒的时间戳，确保全局唯一
- **可读性强**：包含门店和时间信息，便于问题排查
- **业务关联**：直接关联门店，支持按门店维度分析
- **时序性**：时间戳保证了流程的时序关系

#### 2.1.2 核心实现

```typescript
// src/store/trackStore.ts
class TrackStore {
  @observable private processId: string = '';

  /** 生成流程ID */
  @action generateProcessId = (storeCode: string) => {
    this.processId = `${storeCode}${dayjs().format('YYYYMMDDHHmmss')}`;
  };

  /** 设置流程ID (收银台扫码场景) */
  @action setProcessId = (processId: string) => {
    this.processId = processId;
  };

  /** 获取流程ID */
  @computed get getProcessId() {
    return this.processId;
  }
}
```

#### 2.1.3 使用场景

**场景1：员工助手开方**
```typescript
// 进入开方页面时生成
TrackStore.generateProcessId(storeCode);
// processId: F00120260209143045
```

**场景2：收银台扫码**
```typescript
// 使用收银台传入的ID
TrackStore.setProcessId(scanCode);
// processId: 由收银台系统生成，保持一致性
```

**场景3：创智开方**
```typescript
// 创智渠道独立生成
TrackStore.generateProcessId(storeCode);
// processId: CZ00120260209143045
```



### 2.2 Event Track - 事件埋点系统

事件埋点系统基于神策（Sensors Analytics）实现，用于记录用户行为和业务关键节点。

#### 2.2.1 核心事件定义

**1. 问诊资料填写_开始**
```typescript
onTrackConsultationMaterialsStart({
  storeCode: string,        // 门店编码
  processId: string,        // 流程ID
  medicType: string,        // 操作路径-
  internetHospital: string, // 互联网医院
  informationId: string,    // 资料填写单号
  orderCode: string,        // 订单编码
  startTime: string,        // 开始时间

```

**操作路径（medicType）说明：**
- `1`: 员工助手开方（会员详情进入）
- `2`: 员工助手开方（收银扫码进入）
- `3`: 员工助手开方（首页进入）
- `4`: 收银台
- `5`: 收银台处方流转
- `6`: 运营门户视频开方问诊
- `7`: 顾客扫码进入
- `9`: 创智

**2. 问诊资料填写_结束**

onTrackConsultationMaterialsEnd({
  storeCode: string,

  internetHospitalId: string,
  voucherSource: string,    // 复诊凭证来源

**复诊凭证来源（voucherSource）：**

### 2.3 SLS Logger - 阿里云日志服务

基于阿里云SLS（Simple Log Service）实现的日志追踪系统，主要用于IM消息和系统异常的实时监控。

#### 2.3.1 配置信息

```typescript
const slsTracker = new SlsTracker({
  host: 'cn-shenzhen.log.aliyuncs.com',  // 服务入口
  project: 'yf-cashier-log',              // Project名称
  logstore: 'yf-front-log',               // Logstore名称
  time: 6,                                 // 发送间隔（秒）
  count: 3,                                // 批量发送数量
  topic: getStage(),                       // 日志主题（环境）
  source: 'source',
  tags: { tags: 'tags' },
});
```

**环境标识（Stage）：**
- `develop`: 开发环境
- `testing`: 测试环境
- `product`: 生产环境

#### 2.3.2 核心功能

**1. IM消息追踪**
```typescript
slsUploadImLog(sessionId: string, data: any) {
  const pages = getCurrentPages();
  const currentPagePath = pages[pages.length - 1].route;

  slsTracker.send({
    appId: 'medical-im-front',
    level: 'INFO',
    stage: getStage(),
    logTime: Date.now(),
    traceId: sessionId,        // 会话ID作为追踪ID
    thread: distinctId,        // 用户唯一标识
    location: currentPagePath, // 当前页面路径
    message: JSON.stringify({
      ...data,
      eventType: 'IM_MESSAGE_RECEIVE',
      version,                 // 应用版本
    }),
  });
}
```

**2. IM登出追踪**
```typescript
slsUploadKickedOutLog(sessionId: string, data: any) {

    appId: 'medical-im-front',
    level: 'INFO',
    stage: getStage(),
    logTime: Date.now(),
    traceId: sessionId,
    thread: distinctId,
    location: currentPagePath,
    message: JSON.stringify({
      ...data,
      sessionId,
      eventType: 'IM_KICKED_OUT',
      version,
    }),
  });
}
```

#### 4.1.3 持久化策略

**不持久化原因：**
- processId 是会话级别的标识
- 每次新的业务流程都应生成新的ID
- 避免跨会话数据混淆

## 九、问题排查指南

### 9.1 常见问题

#### 9.1.1 ProcessId 为空

**现象：**
```typescript
// 提交时 processId 为空
processId: TrackStore.getProcessId // ''
```

**原因分析：**
1. 未调用 `generateProcessId` 生成
2. 页面刷新导致状态丢失
3. 跨页面传递失败

**解决方案：**
```typescript
// 方案1：在页面加载时生成
public async loadData() {
  if (!TrackStore.getProcessId) {
    TrackStore.generateProcessId(this.storeCode);
  }
}

// 方案2：使用 fallback
const processId = TrackStore.getProcessId || 
  `${storeCode}${dayjs().format('YYYYMMDDHHmmss')}`;

// 方案3：从 consultDetail 获取
const processId = this.consultDetail.processId || 
  TrackStore.getProcessId;
```

#### 9.1.2 埋点数据缺失

**现象：**
神策后台查询不到某些事件数据。

**排查步骤：**
```typescript
// 1. 检查埋点代码是否执行
console.log('埋点执行', eventName, properties);

// 2. 检查神策SDK是否初始化
console.log('Sensors初始化状态', sensors);

// 3. 检查网络请求
// 在浏览器开发者工具中查看网络请求

// 4. 检查环境配置
console.log('当前环境', process.env.TARO_APP_ENV);
console.log('神策URL', getSensorDataUrl());
```

**常见原因：**
1. 埋点代码未执行（条件判断错误）
2. 神策SDK初始化失败
3. 网络请求被拦截
4. 环境配置错误

#### 9.1.3 时间格式不一致

**现象：**
数据分析时发现时间格式不统一。

**问题代码：**
```typescript
// ❌ 错误：多种时间格式

startTime: dayjs().format('YYYY-MM-DD')    // 2026-02-09
startTime: Date.now()                       // 1707465045000
```

**解决方案：**
```typescript
// ✅ 正确：统一使用标准格式
startTime: dayjs().format('YYYY/MM/DD H:mm:ss') // 2026/02/09 14:30:45
```

#### 9.1.4 SLS日志查询不到

**常见原因：**
1. SLS配置错误
